# 内存与存储

## 概念

**内存**是临时存储，用于运行时数据访问；**存储**是永久存储，用于持久化数据。

```
┌──────────────────────────────────────────┐
│              存储金字塔                    │
├──────────────────────────────────────────┤
│  CPU寄存器    速度最快，容量极小    0-cycle│
│  L1/L2/L3缓存                          │ ← CPU核心附近
├──────────────────────────────────────────┤
│  主存(RAM)    速度快，容量GB级      100+  │
├──────────────────────────────────────────┤
│  SSD/磁盘     速度慢，容量TB级      10万+ │
├──────────────────────────────────────────┤
│  网络存储/云   速度最慢，容量无限         │
└──────────────────────────────────────────┘
```

## 关系

**关键连接**：
- 内存 ↔ **CPU**：CPU直接读写内存，延迟低
- 存储 → **内存**：程序启动时从存储加载到内存
- 内存 → **存储**：数据需要持久化时写入存储
- 虚拟内存 → **物理内存**：操作系统通过虚拟内存管理物理内存

**内存墙 (Memory Wall)**：
```
CPU性能提升 (每年~50%)
    vs
内存带宽提升 (每年~10%)
    ↓
差距越来越大 → 缓存变得越来越重要
```

**局部性原理 (Locality)**：
- **时间局部性**：访问过的数据近期可能再次访问
- **空间局部性**：访问的数据附近的数据可能也被访问
```python
# 时间局部性示例
for i in range(1000):
    result = cache[i % 10]  # 反复访问同一数据
# 空间局部性示例
for i in range(1000):
    process(array[i])  # 顺序访问邻接数据
```

**缓存映射方式**：
| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 直接映射 | 每个块对应唯一缓存位置 | 简单，成本低 | 冲突多 |

| 全相联 | 任何位置都可以放 | 最灵活 | 硬件成本高 |

**缓存写策略**：

## 虚拟内存

虚拟内存让程序以为自己拥有连续的内存空间：

```
进程A的虚拟地址空间          物理内存
┌────────────────┐         ┌────────┐
│ 0x0000 - 0xFFFF│ ←映射→  │ 物理页 │
└────────────────┘         └────────┘
```

**优点**：
1. 进程隔离，互不干扰
2. 简化内存管理
3. 支持物理内存扩展

```bash
# Linux 查看虚拟内存信息
cat /proc/meminfo
vmstat 1

# Windows
systeminfo | findstr /C:"Virtual Memory"
```


**缺页异常 (Page Fault)**：访问未映射的页时，OS介入加载对应数据

### 页面置换算法

当物理内存不足时，OS需要将某些页换出到磁盘：

```python
# FIFO (First In First Out) 页面置换
from collections import deque

class FIFOCache:
    def __init__(self, capacity):
        self.capacity = capacity
        self.pages = deque(maxlen=capacity)
        self.page_faults = 0

    def access(self, page):
        if page not in self.pages:
            self.page_faults += 1
            if len(self.pages) >= self.capacity:
                self.pages.popleft()  # 淘汰最老的页
            self.pages.append(page)
            return True  # 缺页
        return False  # 命中

# 示例：访问序列 [1,2,3,4,1,2,5,1,2,3,4,5]，内存容量3
# FIFO缺页率较高，可能产生Belady异常（增加帧数反而缺页增加）
```

```python
# LRU (Least Recently Used) 页面置换
class LRUCache:
    def __init__(self, capacity):
        self.capacity = capacity
        self.cache = {}  # key -> value
        self.access_order = []  # 维护访问顺序

    def access(self, page):
        if page in self.cache:
            self.access_order.remove(page)
            self.access_order.append(page)
            return False  # 命中
        else:
            if len(self.cache) >= self.capacity:
                oldest = self.access_order.pop(0)
                del self.cache[oldest]
            self.cache[page] = page
            self.access_order.append(page)
            return True  # 缺页

# LRU性能好但实现成本高，实际OS常用Clock算法近似LRU
```

```python
# Clock (Second Chance) 页面置换
class ClockCache:
    def __init__(self, capacity):
        self.capacity = capacity
        self.frames = [None] * capacity  # 页帧
        self.use_bits = [0] * capacity     # 使用位(R位)
        self.hand = 0  # 指针位置
        self.page_faults = 0

    def access(self, page):
        # 检查页是否已在内存中
        for i in range(self.capacity):
            if self.frames[i] == page:
                self.use_bits[i] = 1  # 命中则设置使用位
                return False

        # 缺页，用Clock算法选择淘汰页
        while True:
            if self.use_bits[self.hand] == 0:
                # 使用位为0，淘汰该页
                self.frames[self.hand] = page
                self.use_bits[self.hand] = 1
                self.hand = (self.hand + 1) % self.capacity
                self.page_faults += 1
                return True
            else:
                # 使用位为1，清零并继续找
                self.use_bits[self.hand] = 0
                self.hand = (self.hand + 1) % self.capacity

# Clock算法：每个页有使用位，扫描时跳过使用位为1的页并清零，
# 直到找到使用位为0的页淘汰。兼顾性能和实现复杂度。
```

```python
# 页面置换算法对比
algorithms = {
    "FIFO": {"优点": "简单易实现", "缺点": "Belady异常", "性能": "较差"},
    "LRU": {"优点": "性能接近理想", "缺点": "硬件成本高", "性能": "最优"},
    "Clock": {"优点": "近似LRU，成本低", "缺点": "近似精度有限", "性能": "良好"},
    "NFU": {"优点": "实现简单", "缺点": "不考虑访问频率", "性能": "一般"},
    "工作集": {"优点": "动态适应", "缺点": "参数敏感", "性能": "良好"},
}
# Linux使用改良的Clock变体(clock-Pro/WSClock)，结合了Clock和工作集思想
```

**Belady异常**：对于FIFO算法，增加可用帧数反而可能导致缺页率增加，FIFO不遵循局部性原理。

**LFU vs LRU**：LFU基于访问频率，LRU基于最近访问时间。LFU适合频繁使用的内容，但计数器开销大。实际OS多用LRU近似（如Linux的clock算法）。

## 栈 vs 堆

| 特性 | 栈 (Stack) | 堆 (Heap) |
|------|------------|-----------|
| 管理方式 | 编译器自动管理 | 程序员手动管理 |
| 空间大小 | 较小（1-8MB） | 受物理内存限制 |
| 分配速度 | 快 | 慢 |
| 释放方式 | 函数返回自动释放 | free/delete手动释放 |

```c
// 栈 vs 堆示例
void function() {
    int a = 10;           // 栈上，函数返回时自动释放
    int* b = malloc(10);  // 堆上，需要手动free
    free(b);
}
```
