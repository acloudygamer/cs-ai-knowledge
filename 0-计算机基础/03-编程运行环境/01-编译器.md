# 编译器

## 概念

**编译器**是将源代码**翻译**成目标代码（机器码或字节码）的程序。编译的核心是**词法分析 → 语法分析 → 语义分析 → 优化 → 代码生成**的流水线。

```
源代码(.c/.py/.java) → [编译器] → 目标代码 → [链接器] → 可执行文件
                                                    ↓
                                         操作系统加载 → 进程运行
```

## 关系

**关键连接**：
- 编译器 → **CPU**：生成对应指令集（x86/ARM/RISC-V）
- 编译器 → **操作系统**：生成该平台的可执行文件格式（ELF/PE/Mach-O）
- 编译器 → **运行时**：决定生成JVM字节码还是本地机器码
- 解释器 ← **运行时**：解释执行时依赖运行时提供环境
- 编译器 → **中间表示(IR)**：现代编译器通过IR解耦前端和后端

## GCC 编译流程

```bash
# 四步流程：预处理 → 编译 → 汇编 → 链接
gcc -E source.c -o source.i  # 预处理：展开宏、包含头文件
gcc -S source.i -o source.s  # 编译：生成汇编代码
gcc -c source.s -o source.o  # 汇编：生成机器码目标文件
gcc source.o -o program       # 链接：合并目标文件，生成可执行文件

# 一步到位（常用）
gcc -O2 source.c -o program   # -O2 二级优化，-O3 更激进优化，-Os 大小优化
```

### 编译阶段详解

```c
// source.c 示例
#define MAX 100

int square(int x) {
    return x * x;
}

int main() {
    int result = square(MAX);
    return result - MAX;
}
```

**预处理后** (`source.i`)：宏被展开，`#include` 被插入
**汇编后** (`source.s`)：生成汇编指令，如 `imull %edi, %edi; movl %eax, -4(%rbp)`
**目标文件** (`source.o`)：机器码 + 符号表 + 重定位表（未链接的地址）

## LLVM 架构（现代编译器标杆）

```
源代码 → [Clang 前端] → LLVM IR （中间表示）
                                ↓
              ┌─────────────────┼─────────────────┐
              ▼                 ▼                  ▼
        [PASS 1]           [PASS 2]          [PASS N]    ← 优化 passes
              └─────────────────┼─────────────────┘
                                ▼
                    [X86/ARM 后端] → 机器码
```

**LLVM IR** 是编译器的"通用语言"：
- 无限寄存器
- 静态单赋值（SSA）形式
- 平台无关

```bash
# 查看 LLVM IR（通过 Clang）
clang -S -emit-llvm source.c -o source.ll
cat source.ll
```

## 编译器种类

| 编译器 | 特点 | 适用场景 |
|--------|------|----------|
| GCC | 开源，支持多语言多平台 | Linux 默认 |
| Clang/LLVM | 模块化设计，诊断信息友好 | macOS Xcode |
| MSVC | Windows 原生，优化好 | Visual Studio |
| javac | Java 专用 | Java 开发 |
| rustc | Rust 自带，ML 系 borrow checker | Rust 开发 |

## 静态链接 vs 动态链接

```bash
# 静态链接：库代码直接打包进可执行文件
gcc -static source.c -o program    # 文件大（MB级），但部署简单

# 动态链接：运行时加载共享库
gcc source.c -o program           # 文件小（KB级），共享库可更新

# 查看可执行文件的动态依赖
ldd program                       # Linux
otool -L program                  # macOS
dumpbin /dependents program       # Windows
```

**动态库在 Linux 上是 `.so` 文件，Windows 上是 `.dll`，macOS 上是 `.dylib`。**

**动态链接优缺点**：
- 优点：可执行文件小、共享库更新不需要重新编译、节省内存（多个进程共享）
- 缺点：运行时依赖库、可能遇到" DLL hell "、部署复杂

## 交叉编译

在一台机器上编译另一平台的可执行文件：

```bash
# ARM 交叉编译（从 x86 编译 ARM 程序）
aarch64-linux-gnu-gcc source.c -o program

# macOS 上交叉编译 Linux 二进制
x86_64-linux-gnu-gcc source.c -o program

# Windows 上交叉编译 Linux
x86_64-linux-gnu-gcc source.c -o program
```

**为什么需要交叉编译**：
- 嵌入式设备（ARM/MIPS）资源有限，无法本地编译
- 开发者在强大机器上编译，部署到多个目标平台
- 加速编译（编译服务器远比目标设备快）

## 编译优化等级

| 选项 | 优化内容 | 编译时间 | 适用场景 |
|------|----------|----------|----------|
| `-O0` | 无优化 | 最快 | 调试 |
| `-O1` | 基本优化，减小代码大小 | 快 | 平衡 |
| `-O2` | 激进优化，不牺牲调试 | 中等 | 生产默认 |
| `-O3` | +向量化、函数内联 | 慢 | 性能关键 |
| `-Os` | 优先减小大小 | 中等 | 嵌入式 |
| `-Ofast` | +浮点优化（违反标准） | 慢 | 科学计算 |
